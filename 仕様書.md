# pi-multi-agent 仕様書 v2.2.0
> pi（対話/非対話）＋ tmux ＋ pi-gl-mem によるパイプライン型マルチエージェントシステム。
> manager が役割設定・経路設定を行い、worker 同士が連鎖して処理し、最終結果のみ manager に報告する。

## 1. 全体の流れ（起動→初回タスク完了）

```bash
bash pi-team-run.sh
  ↓ 対話入力: チーム名, ワーカー数
  ↓
pi-team-run.sh: ディレクトリ生成→MEMORY.md配布→tmux N+1ペイン作成→manager(pi対話)起動
  ↓
manager: 「これから各 worker の役割を設定します。w01 の役割は？」
  ↓
ユーザー: 「設計」「実装」「テスト」…と各 worker の役割を順に回答
  ↓
manager: 各 worker の .pi-gl-mem/MEMORY.md に役割を write で追記
  ↓
manager: 「処理の経路（順番）を教えてください」
  ↓
ユーザー: 「w01 → w02 → w03」
  ↓
manager: 各 worker の MEMORY.md に経路情報を write で追記
  ↓
manager: 最初の worker 用の指示ファイルを作成（タスク＋経路情報を含む）
  ↓
manager → tmux send-keys で w01 を pi -p 起動
  ↓
w01: 指示を実行 → 成果を w02/inbox/ に作成 → tmux send-keys で w02 起動
  ↓
w02: 同様 → w03 へ
  ↓
w03（最終）: 完了 → manager に最終報告（tmux send-keys）
  ↓
manager: 報告を受け取りユーザーに提示
```

**ペイン構成（WORKER_N=3 の例）**:
```
┌─────────────────┬─────────────────┐
│ manager(pi対話)  │ worker02(待機)  │
├─────────────────┼─────────────────┤
│ worker01(待機)  │ worker03(待機)  │
└─────────────────┴─────────────────┘
```

## 2. 概要

2層構造 + パイプライン連携。manager が設定を行い、worker 同士が直接連鎖する。

| 層 | エージェント | モード | 役割 |
|----|------------|--------|------|
| 上位 | **manager** | `pi`（対話） | 役割設定・経路設定・最初のトリガー・最終報告受領 |
| 下位 | worker01〜04 | `pi -p` | 指示された作業を実行し、次の worker に連鎖（最終の場合は manager に報告） |

**基本原則**:
- manager は自分では作業しない。設定と管理のみ
- worker 同士が tmux send-keys で直接連鎖する（パイプライン）
- 最終結果のみ manager に届く
- 各 worker の指示ファイルに経路情報（次 worker／次ペイン）が含まれる
- worker は bash ツールで tmux send-keys を実行し、次の worker を起動
- 各エージェントの `.pi-gl-mem/` に記憶が蓄積される

**ディレクトリ構成**:
```
pi-multi-agent/
├── pi-team-run.sh / 仕様書.md
├── oldfiles/
└── tasks/<team_name>/
    ├── manager/inbox/ outbox/ .pi-gl-mem/       # pi 対話
    ├── worker01/inbox/ outbox/ .pi-gl-mem/       # pi -p
    ├── worker02/ .../
    ├── worker03/ .../
    └── worker04/ .../
```

## 3. 画面構成

ペイン数は WORKER_N + 1（manager + worker 数）。起動時は manager のみ pi 対話モード、他は bash 待機。

**ペイン番号マップ**: manager=0, worker01=1, worker02=2, worker03=3, worker04=4

**tmux 起動手順**（シンプル版）:
```bash
tmux new-session -d -s "<session>" -c "tasks/<team>/manager" -n agents
for i in $(seq 1 $WORKER_N); do
  tmux split-window -h -c "tasks/<team>/worker${i}"
done
tmux select-layout tiled
```

ログ出力機能は削除済み。`pipe-pane` / `tail -f` は使用しない。

## 4. エージェント定義

### 4.1 manager（pi 対話モード・常駐）

**役割**: 役割設定・経路設定・最初のトリガー・最終報告受領。自分では作業しない。

**最重要ルール**:
- 自分ではコードを書かない、調査しない
- ただし worker の役割・経路設定として MEMORY.md を編集するのは許可

**ルール**:
```
1. 各 worker の役割をユーザーに順に聞く（「w01 の役割は？」→「w02 の役割は？」…）
2. 各 worker の .pi-gl-mem/MEMORY.md に write ツールで役割を追記
3. 処理の経路（順番）をユーザーに聞く（例: 「w01 → w02 → w03」）
4. 各 worker の .pi-gl-mem/MEMORY.md に write ツールで経路を追記（次 worker／次ペイン）
   最終 worker には「次: なし」を設定
5. 最初の worker 用の指示ファイルを作成（タスク + 経路情報を含む）
6. 最初の worker を起動: tmux send-keys -t {session}:agents.1 "pi -p 'inbox/{ファイル}'" Enter
7. 以降は最終報告が来るまで待機
8. write_local_memory target=daily
```

### 4.2 worker（pi -p・呼ばれたら起動・終了）

**役割**: 指示された作業を実行し、経路に従い次の worker に連鎖（最終の場合は manager に報告）。

**テンプレート MEMORY.md**:
```markdown
## 役割
（manager が設定）

## 経路
次:
次ペイン:

## ルール
1. inbox/ の指示ファイルを読んで着手
2. 指示ファイルのタスクを実行し成果物を作成
3. 完了後、指示ファイル内の経路情報を確認:
   - 「次: (worker名)」「次ペイン: (番号)」があれば
     → 成果ファイルを ../{次}/inbox/ に作成
     → bash で tmux send-keys -t {session}:agents.{次ペイン} "pi -p 'inbox/{ファイル}'" Enter を実行
   - 「次: なし」なら manager に最終報告
     → 報告ファイルを ../manager/inbox/ に作成
     → bash で tmux send-keys -t {session}:agents.0 "[{short} 完了] 最終報告" Enter を実行
4. write_local_memory target=daily
```

## 5. ファイル通信プロトコル

**ファイル名**: `{TS}_{short}_to-{相手}_{件名}.md`

| 略称 | mgr | w01 | w02 | w03 | w04 |
|------|-----|-----|-----|-----|-----|
| 対象 | manager | worker01 | worker02 | worker03 | worker04 |

**通知フロー**:
- **manager → 最初の worker**: `../worker01/inbox/` に指示ファイル作成 → `tmux send-keys "pi -p 'inbox/{ファイル}'" Enter`（timeout なし）
- **worker → 次の worker**: `../{次}/inbox/` に成果ファイル作成 → `tmux send-keys "pi -p 'inbox/{ファイル}'" Enter`（bash ツール経由）
- **最終 worker → manager**: `../manager/inbox/` に報告ファイル → `tmux send-keys "[完了] 最終報告" Enter`

**パス注意**: pi は自ディレクトリ内で起動。他エージェントへのパスは `../{相手}/inbox/`。

## 6. ワークフロー

```
ユーザー ←→ manager(対話)
               │ 設定・トリガー
               ▼
          w01 → w02 → w03 → manager（最終報告）
          (pi -p) (pi -p) (pi -p)
```

**パターン例**:
1. **単一worker**: manager → w01 → manager
2. **パイプライン**: manager → w01(設計) → w02(実装) → w03(テスト) → manager
3. **複数ライン**: manager → [w01→w02] と [w03→w04] の2ライン並列

## 7. 起動方法

```bash
cd /mnt/c/projects/pi-multi-agent
bash pi-team-run.sh
  → チーム名を入力
  → ワーカー数 (1-4) を入力
  → tmux セッションが起動、manager が役割設定を開始
```

内部で `pi-team-run.sh` が以下を実行:
1. `tasks/<team_name>/` ディレクトリ生成 + 各 .pi-gl-mem 初期化
2. MEMORY.md テンプレート書き込み（manager + worker01〜04）
3. tmux N+1ペイン作成 + tiled レイアウト
4. manager(pi対話)起動＋最初の役割設定質問
5. Windows Terminal別窓表示（WSL）

**終了**: tmux セッションは残る（`tmux kill-session -t <team_name>` で完全終了）

## 8. 学習と知識蓄積

各エージェントの `.pi-gl-mem/` にプロジェクト知識が蓄積。再起動時に自動ロードされる。
- **manager**: 役割設定・経路・要件・決定事項
- **worker**: 作業履歴・成果物

## 9. 過去の変更履歴

| バージョン | 変更内容 |
|-----------|---------|
| v2.2.0 | パイプライン方式に変更。worker 同士が連鎖。system-assist/worker00 廃止。worker01 開始 |
| v2.1.0 | ペイン作成をループ+select-layout tiled に簡素化。ログ機能削除。timeout 削除 |

### v2.1.0 → v2.2.0 の主な変更点
- `AGENTS=(manager worker00 worker01 worker02 worker03 worker04)` → `AGENTS=(manager worker01 worker02 worker03 worker04)`
- system-assist / worker00 を廃止し、worker01 から開始
- manager が役割設定と経路設定を担当
- worker 完了後は manager ではなく次の worker に連鎖
- 最終 worker のみ manager に報告
- worker は bash ツールで tmux send-keys を実行

### v2.0.0 → v2.1.0 の主な変更点
- ペイン作成をループ + select-layout tiled に簡素化
- `mkdir -p "$TEAM_DIR/logs"` / `tmux pipe-pane` / `tail -f` 削除
- `teams/` → `squads/` → `tasks/`（NTFS case-sensitivity 問題回避）
- `timeout 180` を削除（タイムアウト方式から通知方式に変更）

## 11. 設計思想

| 原則 | 説明 |
|------|------|
| パイプライン | worker 同士が連鎖。中間結果は manager を経由せず直接次の worker へ |
| 設定ファースト | 起動後まず役割と経路を設定。その後タスク実行 |
| 最小介入 | manager は最初のトリガーのみ。後は最終報告を待つ |
| 学習継続 | pi-gl-mem で知識蓄積。プロジェクトを跨いでも活用 |
| 動的構成 | ワーカー数 1〜4 で可変。ペイン数も自動調整 |
| timeout 不要 | 待ち時間を設定せず、完了通知で駆動 |
