# pi-multi-agent 仕様書
> マルチエージェント連携システム — pi + tmux + pi-gl-mem による最小構成（257行の単一スクリプト + JSON 設定）
---

## 1. 概要
複数の AI エージェント（pi）が tmux ペイン上で協調動作し、ファイル通信とリアルタイム通知によりワークフローを実行する。
- **pi -p**（非対話モード）で各エージェントを起動。処理後にプロセス終了
- **tmux** で3ペイン縦分割。`send-keys` で次エージェントを即時呼び出し
- **ファイル通信**: 各エージェントの `inbox/` `outbox/` に Markdown ファイル
- **pi-gl-mem**: 各エージェントが `.pi-gl-mem/` に長期記憶・日次ログを保持
---

## 3. pi-team-run.sh 解説
### 3.1 全体構成
3セクションで構成される。
```
行  1- 18: プロローグ（シバン・定数・パス解決）
行 20- 38: デフォルトエージェント定義
行 40- 57: 関数定義（gen_memory）
行 60-181: RUN モード（JSON → チーム起動・実行）
行 183-257: CREATE モード（JSON 設定ファイル生成）
```
実行の流れ:
```
bash pi-team-run.sh team_a
  │
  ├── team_a.json が存在する？
  │   ├── YES → RUN モード
  │   └── NO  → CREATE モード → JSON生成 → exec で RUN モードへ
  │
  ├── teams/team_a/ が存在する？
  │   ├── NO  → 各エージェントディレクトリ + MEMORY.md を生成
  │   └── YES → 再利用（.pi-gl-mem は保持）
  │
  ├── タスクファイル投入（タイムスタンプ付き、JSON の task_template から読込）
  ├── tmux セッション作成（3ペイン縦分割）
  ├── 自動トリガー送信
  ├── Windows Terminal で別窓表示
  └── ログ tail + 完了検出ポーリング
```
### 3.2 プロローグ（行1-20）
```bash
BASE_DIR="$(cd "$(dirname "$0")" && pwd)"     # スクリプトの絶対パス
TEAM_NAME="${1:?Usage: ...}"                   # 第1引数: チーム名
CONFIG="$BASE_DIR/$TEAM_NAME.json"             # 設定ファイル
TEAM_DIR="$BASE_DIR/teams/$TEAM_NAME"          # データディレクトリ
WT=$(which wt.exe 2>/dev/null || true)          # Windows Terminal 検出
TS=$(date +%Y%m%d_%H%M)                        # この実行のタイムスタンプ
```
- `TS` は全ファイル名に使われ、実行を識別する
- `WT` → `cmd.exe /c start` → `tmux attach` の順にフォールバック
### 3.3 gen_memory()（行22-38）
各エージェントの `.pi-gl-mem/MEMORY.md` を書き出す。`{TS}` `{short}` を実行時に置換。
```bash
gen_memory "$name" "$role" "$SHORT" "${RULES[@]}"
# $1: エージェント名 (agent1_editor)
# $2: 役割説明
# $3: 略称 (a1 / a2 / a3)
# $4..$n: ルール配列 → MEMORY.md に番号付きリストで出力
```
### 3.4 RUN モード（行39-167）
**JSON パース**: `jq` で session / timeout / workflow / loops / agents[].rules を読む。
**ディレクトリ生成**（初回のみ）:
1. `mkdir -p teams/team_a/{agent}/`
2. pi-gl-mem 初期化
3. `jq` からルール配列を読んで gen_memory
4. `mkdir -p inbox outbox`
5. `.gitignore` に `.pi-gl-mem/` 追加
**タスク投入**: `jq -r '.task_template' "$CONFIG" > "$TASK_FILE"`
- ファイル名: `{TS}_from-user_to-{FIRST}_タスク.md`
- 毎回新規作成（古いファイルは履歴として残る）
**tmux セットアップ**:
```bash
tmux new-session -d -s "writing-team" -c "teams/team_a/agent1_editor" -n agents
tmux pipe-pane ... "cat >> logs/agent1_editor.log"
tmux split-window -v -c "teams/team_a/agent2_writer"
tmux split-window -v -c "teams/team_a/agent3_reviewer"
```
ペイン構造:
```
┌─────────────────┐
│ agent1_editor   │  ペイン0
├─────────────────┤
│ agent2_writer   │  ペイン1
├─────────────────┤
│ agent3_reviewer │  ペイン2
└─────────────────┘
```
**自動トリガー**: `tmux send-keys` で editor に具体的なファイル名を指定して起動。
**外部ターミナル**: `wt.exe wsl tmux attach` → `cmd.exe /c start` → `tmux attach`
**ログ表示**:
```bash
tail -f logs/*.log | stdbuf -oL sed 's/\x1b\[[0-9;]*[a-zA-Z]//g; s/\r//g'
```
**完了検出**: 10秒間隔で最大30回（5分）ポーリング。2段階で検出する。
1. **優先**: ファイルの先頭行に `✅ ワークフロー完了` を含むファイルを検出（エージェントがルール6に従い出力するマーカー）
2. **フォールバック**: 上記が無ければファイル名に `*承認*` または `*合格*` を含むファイルを検出
- 検出時: バナー表示 + 生成物一覧 → `exit 0`
- タイムアウト: `⏰ タイムアウト` → `exit 1`
### 3.5 CREATE モード（行183-257）
`team_a.json` が存在しない場合、`DEFAULT_AGENTS` 配列から Python で自動生成。
```python
# Python ヒアドキュメント
for i, (name, role) in enumerate(agent_defs):
    is_final = (i == 0)      # 最初のエージェントが最終承認者
    rules = [ ... ]
    agents.append({"name": name, "role": role, "rules": rules})
config = {
    "session": ..., "timeout": ..., "workflow": ...,
    "loops": [...], "task_template": "...", "agents": agents
}
```
生成後、`exec bash "$0" "$TEAM_NAME"` で自身を RUN モードとして再起動。

## 4. team_a.json 解説
### 4.1 構造
```json
{
  "session": "writing-team",
  "timeout": 180,
  "workflow": "agent1_editor(0)→agent2_writer(1)→agent3_reviewer(2)→agent1_editor(0)",
  "loops": [{"between": ["agent3_reviewer", "agent2_writer"], "min": 1, "max": 3}],
  "task_template": "# タスク\n\nCLI ツール greet を作成してください。\n...",
  "agents": [ { "name": "...", "role": "...", "rules": [...] }, ... ]
}
```
### 4.2 トップレベルフィールド
| フィールド | 型 | 説明 |
|-----------|-----|------|
| `session` | 文字列 | tmux セッション名 |
| `timeout` | 数値 | pi のタイムアウト秒数（`timeout {n} pi -p`） |
| `workflow` | 文字列 | エージェント間の連鎖（AI が次を判断） |
| `loops` | 配列 | 差し戻しループ制限（between / min / max） |
| `task_template` | 文字列 | タスク内容（Markdown）。実行ごとにタイムスタンプ付きファイルとして投入 |
| `agents` | 配列 | 各エージェントの定義 |
### 4.3 agents 配列
各エージェントは3フィールド:
```json
{
  "name": "agent1_editor",
  "role": "あなたは編集者（ディレクター）です。...",
  "rules": [ "ルール1", "ルール2", ..., "ルール8" ]
}
```
| フィールド | 説明 |
|-----------|------|
| `name` | ディレクトリ名・ペイン番号マップに使用。short は自動で `a1`～`a3` |
| `role` | MEMORY.md 先頭に書かれる役割説明。pi のコンテキスト |
| `rules` | MEMORY.md に番号付きリストで書き込まれるルール。`{TS}` `{short}` は実行時に置換 |
### 4.4 エージェント一覧
| エージェント | 略称 | ペイン | 役割 | ルール数 |
|------------|------|--------|------|---------|
| `agent1_editor` | a1 | 0 | タスク分解・指示・最終承認 | 8（最終承認ルール含む） |
| `agent2_writer` | a2 | 1 | 実装・レビュー提出 | 7 |
| `agent3_reviewer` | a3 | 2 | レビュー・合格/差し戻し | 7 |
### 4.5 MEMORY.md ルール一覧
**ルール1**: inbox 監視
```
inbox/ にファイルが来たら読んで着手
```
**ルール2**: ファイル通信
```
完了後: 自分の outbox/ と相手の inbox/ にファイル作成。
ファイル名: {TS}_{short}_to-{相手}_件名.md。パス: ../{相手}/inbox/
```
- `../{相手}/inbox/` — pi は自ディレクトリ内で起動するため
**ルール3**: tmux 通知
```
通知: tmux send-keys -t writing-team:agents.{相手ペイン}
      "timeout 180 pi -p 'inbox/{ファイル名} を処理'" Enter
```
**ルール4**: ワークフロー
```
ワークフロー: agent1_editor(0)→agent2_writer(1)→agent3_reviewer(2)→agent1_editor(0)
```
**ルール5**: ループポリシー（該当エージェントのみ）
```
ループポリシー: agent3_reviewer↔agent2_writer: 最低1回、最大3回まで繰り返し
```
**ルール6**（最終承認者のみ）: 完了表示
```
【厳守】ワークフロー完了時（自分が最終承認者）:
応答の先頭行に "=== ✅ ワークフロー完了 ===" のみを出力せよ。
「以上で完了しました」などの言い換えは一切禁止。
完了報告は2行目以降に書け
```
**ルール7**: ユーザー対応
```
ユーザーはtmux外。連絡はinbox/へのファイル作成
```
**ルール8**: ログ
```
ログ: write_local_memory target=daily
```
### 4.6 プレースホルダー
| プレースホルダー | 置換後（例） | 置換元 |
|----------------|-------------|--------|
| `{TS}` | `20260625_1434` | スクリプト起動時の `$TS` |
| `{short}` | `a1` | gen_memory の第3引数 |
`{相手}` `{相手ペイン}` `{ファイル名}` は AI が文脈から置き換える指示語。スクリプトは解釈しない。

## 5. ファイル通信プロトコル
### 5.1 ファイル名形式
```
{TS}_{short}_to-{相手}_{件名}.md
```
例:
```
20260625_1434_a1_to-agent2_writer_greet作成指示.md
20260625_1434_a2_to-agent3_reviewer_greetレビュー依頼.md
20260625_1434_a3_to-agent1_editor_greet承認報告.md
```
### 5.2 通知フロー
1. エージェントA 完了 → `../{B}/inbox/` にファイル作成
2. `tmux send-keys -t {SESSION}:agents.{ペイン} "pi -p 'inbox/{ファイル名}'"`
3. エージェントB が起動・処理
### 5.3 パスの注意
pi は各エージェントのディレクトリ内で起動する。他エージェントへのパスは `../{相手}/inbox/` と記述する（`{相手}/inbox/` は自ディレクトリ配下になる）。

## 6. ワークフロー
```
ユーザー
  │
  ▼ (タスクファイル)
agent1_editor (ペイン0) ──指示──→ agent2_writer (ペイン1)
  ▲                              │
  │                              ▼  (レビュー依頼)
  │                          agent3_reviewer (ペイン2)
  │                              │
  └──── 承認報告 ────────────────┘
```
### ループポリシー
- `agent3_reviewer ↔ agent2_writer`: 最低1回、最大3回まで繰り返し
- 修正が必要なら reviewer → writer に差し戻し
---

## 7. 起動方法
```bash
# 初回（チーム生成＋起動）
bash pi-team-run.sh team_a
# 再実行（新しいタスク投入、古いファイルは履歴）
bash pi-team-run.sh team_a
# Ctrl+C でログ表示終了（tmux は継続）
```
### 動作の流れ
1. `team_a.json` なければ CREATE モードで生成
2. `teams/team_a/` なければ各エージェントディレクトリ作成
3. pi-gl-mem 初期化 + MEMORY.md 書き込み
4. `task_template` からタスクファイルを Editor の inbox に投入
5. tmux 3ペイン作成
6. Editor に自動トリガー（具体的なファイル名指定）
7. Windows Terminal で別窓表示
8. 元ターミナルでログ tail
9. 承認ファイル検出で完了バナー表示
---

## 8. 完了検出
### 表示
```
╔════════════════════════════════════════╗
║      ✅ ワークフロー完了！              ║
╚════════════════════════════════════════╝
  完了ファイル: 20260625_1434_a3_to-a1_greet承認報告.md
  生成物: teams/team_a/greet
  通信: agent1_editor/outbox/... agent2_writer/outbox/...
```
### 検出条件（2段階）
1. **優先**: ファイルの先頭行に `✅ ワークフロー完了` が含まれること（エージェントがルール6に従い出力する完了マーカー）
2. **フォールバック**: 上記が無ければ、ファイル名に `*承認*` または `*合格*` を含むこと
- タスクファイルより**新しい**ファイルのみ（古い実行は無視）
- 10秒間隔 × 最大30回（5分）ポーリング

## 9. カスタマイズ
### タスク内容を変える
```bash
jq '.task_template = "新しいタスク"' team_a.json > tmp && mv tmp team_a.json
```
### エージェントの役割を変える
`DEFAULT_AGENTS` 配列の `:` 以降を書き換え。`team_a.json` を削除して再生成。
### タイムアウト・セッション名
`team_a.json` の `timeout` `session` を直接編集。
### 別チーム
`team_b.json` を複製しセッション名を変更:
```bash
bash pi-team-run.sh team_b
```
### エージェント追加
`DEFAULT_AGENTS` に要素を追加し、Python の agent_defs とループ設定を調整。

## 11. 設計思想
| 原則 | 説明 |
|------|------|
| **最小構成** | スクリプト1個(257行) + JSON設定1個、依存最小限 |
| **自己完結** | JSON に全ルール。スクリプトにルールロジックは持たない |
| **非永続プロセス** | 全エージェント `pi -p`（処理後終了） |
| **ファイルベース通信** | inbox/outbox の Markdown。シンプルで追跡容易 |
| **タイムスタンプ識別** | 全ファイルに実行 TS。履歴と現実行を区別 |
| **学習継続** | `.pi-gl-mem/` は保持。再実行時に過去の記憶を参照 |
| **クロスプラットフォーム** | WSL(Windows Terminal) / macOS / Linux |
