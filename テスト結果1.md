## テスト内容

### トリガー方法
全テストにおいて、ユーザー（この pi セッション）から以下の手順で最初の起動を行った。

**1. 初期タスク作成** — editor/inbox/ にファイルを置く
```
cat > editor/inbox/2026-06-25-1200_from-user_to-editor_最初のタスク.md << 'EOF'
# 最初のタスク
...
EOF
```

**2. tmux send-keys で editor に指令**
```
tmux send-keys -t writing-team:agents.0 \
  "cd /path/to/editor && pi -p 'inbox を確認して最初のタスクを処理してください'" Enter
```

※ 1回目（pi常駐版）では `sleep 2` を挿入してから送信していたが、これは pi の起動完了を待つため。

### エージェント間の連鎖
各エージェントの MEMORY.md に従い、完了後に以下を自動実行：
1. 自分の `outbox/` と相手の `../{相手}/inbox/` にファイル作成
2. `tmux send-keys -t writing-team:agents.{相手ペイン} "pi -p 'inbox/{ファイル名} を処理'" Enter`

### タスク
シンプルな CLI ツール（`hello` スクリプト）を作成する。

**仕様**
- 引数なし: `Hello, World!` と出力
- 引数あり: `Hello, <名前>!` と出力
- 言語: Bash

### ワークフロー
```
ユーザー(このpi) → editor(0) → writer(1) → reviewer(2) → editor(0) へ承認報告
```

## タイムアウト対策
- `timeout 180 pi -p '...'` で各エージェントをラップ
- 3分（180秒）経過で強制終了、bash プロンプトに戻る
- 数値を変えれば調整可能（MEMORY.md の1箇所）
- タイムアウト後は次の send-keys が届けば再度起動する

## 確認された問題点

### 1. 相対パスの誤り（1回目）
- MEMORY.md が `writer/inbox/` と記載していたため、editor が `editor/writer/inbox/` にファイルを作成してしまった
- **対策**: `../writer/inbox/` に修正済み

### 2. tmux ペインインデックスのズレ（検証時）
- `split-window -h` でペイン番号が期待と異なることがあった
- **対策**: 縦3段（`split-window -v` のみ）に変更して固定

### 3. 古いファイルの混在（2回目）
- 1回目の残骸ファイルが inbox に残っており、エージェントが古いファイルも処理した
- **対策**: 運用前に `inbox/` `outbox/` をクリーンアップする必要あり

### 4. パス指定の冗長性（2回目）
- 各ペインは既に正しいディレクトリで起動しているため、`cd ../{agent}` は不要（単に `pi -p` でOK）
- ただし初回起動時は明示的な cd が必要

### 5. sleep の要否
- **pi常駐版**では `sleep 2` が必要だった（pi がプロンプトを表示するまで待つため）
- **pi -p 版では sleep は不要**。理由：
  - bash ペインは常にプロンプト待ち状態
  - `tmux send-keys` は入力バッファに一瞬で書き込む
  - 仮に前のコマンド実行中でも、バッファに溜まり bash が空いた時に自動実行される
- 実際、2回目のテストではエージェント間連鎖に一切 sleep を使っていない

### 6. sleep を排除する仕組みについて
**既に排除済みである**。`run-team.sh` には sleep は一切含まれていない。
起動 → 即座に `tmux send-keys` でトリガー可能。

なお初回トリガーも sleep 不要であり、以下で即座に起動できる：
```
# tmux 起動（bash run-team.sh teams/team_a）
# 直後にトリガー（sleep 不要）
tmux send-keys -t writing-team:agents.0 \
  "cd teams/team_a/editor && pi -p 'inbox を確認して処理'" Enter
```

## 結論

- **マルチエージェント連携は正常に動作**
- **pi -p（非対話モード）**により、常駐プロセスを残さずにエージェント間連鎖が可能
- MEMORY.md に役割とルールを書けば、AI がルールに従って行動する
- ファイル連携（inbox/outbox）+ tmux send-keys 通知の組み合わせで十分実用的
